# 320. 백로그 리파인먼트 (Backlog Refinement)

이 문서는 2026-03 기준 LessonRing Backend의 현재 구현 상태, 문서 상태, 최근 커밋 흐름을 바탕으로 백로그를 정리한 문서다.

목표:

- 우선순위를 현재 프로젝트 상태에 맞게 재정렬한다.
- 큰 작업을 실제 실행 가능한 단위로 쪼갠다.
- 무엇이 완료되었고 무엇이 후속 과제인지 명확히 한다.
- 팀이 바로 스프린트 후보로 가져갈 수 있는 수준까지 할 일 목록을 구체화한다.

우선순위 기준:

- 1차 기준: 최근 커밋 날짜 흐름
- 2차 기준: 운영 리스크
- 3차 기준: 정합성/테스트 공백
- 4차 기준: 기능 확장 가치

정렬 원칙:

- 최근 커밋에서 막 정리된 영역은 "후속 안정화" 과제로 둔다.
- 오래된 구현이지만 운영 검증이 약한 영역은 상위 우선순위로 올린다.
- 아직 구현 흔적만 있고 운영 가치가 낮은 확장 기술은 후순위로 둔다.

---

# 1. 최근 커밋 흐름 기반 프로젝트 상태

최근 커밋 흐름을 보면 프로젝트는 다음 순서로 진화했다.

## 1.1 2026-03-14 ~ 2026-03-18

주요 흐름:

- 코어 도메인 1차 구현
- 코어 도메인 2차 고도화
- 전체 흐름 안정화
- 결제/이용권/예약 환불 흐름 구현
- Swagger, 예외 처리 보강

해석:

- 이 시기에는 제품의 기본 도메인 흐름을 세우는 작업이 핵심이었다.

## 1.2 2026-03-24

주요 흐름:

- 결제 안정성 강화
- webhook 멱등 처리
- refund 멱등성 및 Redis 락 적용
- 로드맵, 비전/지표, 다국어 설계 문서 추가

해석:

- 이 시기부터 단순 기능 구현보다 운영 정합성과 확장 설계가 중요해졌다.

## 1.3 2026-03-25 ~ 2026-03-26

주요 흐름:

- approve 2차 구현
- webhook 안정화
- 교차 충돌 대응
- 테스트 케이스 및 QA 문서 정리
- 시스템/도메인 아키텍처 문서 추가
- 결제 안정성과 테스트 문서 최신화

해석:

- 현재 가장 성숙한 영역은 `payment`
- 상대적으로 다음 단계로 넘어가야 할 영역은 `운영 검증`, `analytics`, `관리자 조회`, `배포/관측`

# 2. 현재 백로그 재정렬 원칙

현재 단계에서 중요한 것은 새 기능을 무작정 늘리는 것이 아니다.

우선순위 원칙은 다음과 같다.

1. 운영 정합성 보강
2. 테스트와 실행 환경 검증
3. 관리자/운영 조회 기능
4. 분석/로그 체계 구축
5. 국제화 및 메시지 체계
6. 외부 연동 확장
7. 인프라/배포 고도화

즉, 최근 커밋 순서를 반영하되 단순히 "최근에 한 일의 다음 일"이 아니라, 현재 코드가 가장 약한 지점을 먼저 메우는 방향으로 백로그를 정리한다.

# 3. 백로그 우선순위

## P0. 즉시 처리 우선

운영 리스크가 커서 가장 먼저 다뤄야 하는 항목이다.

### BL-001. `local`, `dev` 실행 검증 자동화

- 배경: 로드맵 문서에서 최우선 과제로 반복 등장
- 근거: 최근 커밋은 결제 안정성에 집중했지만 실행 환경 검증은 아직 문서 중심
- 해야 할 일:
  - `local` 프로필 bootRun 검증
  - `dev` 프로필 설정값 검증
  - DB migration 성공 여부 검증
  - Redis 연결 검증
  - 실행 가이드 최신화
- 완료 기준:
  - 신규 개발자가 문서만 보고 `local` 실행 가능
  - `dev` 환경에서 핵심 API 구동 가능

### BL-002. 결제 운영 점검 항목 마무리

- 배경: `payment-follow-up-status.md` 기준 일부 미완료 항목 존재
- 해야 할 일:
  - webhook secret 운영 주입 방식 확정
  - Redis 운영 설정 기준 확정
  - 결제/웹훅 운영 모니터링 항목 정의
  - 운영 알림 기준 정의
- 완료 기준:
  - 결제 관련 환경 변수/운영 기준 문서화 완료
  - 운영 중 결제 장애 시 확인 절차가 존재

### BL-003. 예약/출석 로그 보강

- 배경: 결제 로그는 강해졌지만 예약/출석 로그는 상대적으로 약함
- 해야 할 일:
  - 예약 생성/취소 로그 추가
  - 출석 처리/취소 로그 추가
  - no-show 배치 로그 세분화
  - requestId 기반 연결성 준비
- 완료 기준:
  - 예약/출석 주요 성공/실패 흐름이 로그로 추적 가능

## P1. 단기 스프린트 우선

이미 구현된 도메인을 운영 가능한 수준으로 끌어올리는 작업이다.

### BL-004. 핵심 도메인 테스트 확대

- 배경: payment 테스트는 강화되었지만 booking/attendance/membership은 상대적으로 약함
- 해야 할 일:
  - BookingService happy path / invalid path 보강
  - 중복 예약 방지 테스트 보강
  - AttendanceService 차감/복구 시나리오 추가
  - Membership 상태 전이 테스트 추가
- 완료 기준:
  - 예약/출석/회원권 핵심 규칙 테스트 보유

### BL-005. 정합성 점검 SQL 및 운영 체크리스트 도입

- 배경: 현재 운영 리스크는 "처리는 되었는데 상태가 틀린 경우"
- 해야 할 일:
  - 결제 완료 후 회원권 미생성 SQL 점검
  - 환불 후 회원권/예약 상태 점검 SQL 추가
  - 출석 후 회원권 상태 점검 SQL 추가
  - 정기 점검 체크리스트 문서화
- 완료 기준:
  - 운영자가 이상 상태를 수동 점검할 수 있음

### BL-006. `analytics` 조회 API 1차 구현

- 배경: analytics 문서와 설계는 존재하지만 실제 서비스 구현은 TODO
- 해야 할 일:
  - `overview` API 구현
  - `integrity` API 구현
  - `payments` API 구현
  - Query Repository 작성
- 완료 기준:
  - 운영 핵심 수치를 API로 조회 가능

### BL-007. 로그 표준 실제 적용

- 배경: logging standard 문서는 생겼으나 코드 전반 적용 전
- 해야 할 일:
  - requestId 필터 도입
  - access log 추가
  - 결제 로그에 duration 정리
  - 예약/출석 로그 패턴 통일
- 완료 기준:
  - 주요 API 요청과 핵심 도메인 흐름 추적 가능

## P2. 구조 개선 우선

단기 운영 리스크보다 한 단계 뒤지만, 앞으로 기능 확장 전 정리하면 좋은 항목이다.

### BL-008. Payment 엔티티 및 서비스 책임 재정리

- 배경: 이미 관련 리팩터링 문서가 존재
- 해야 할 일:
  - Payment 엔티티 책임 점검
  - Service 과도 책임 분리
  - 상태 전이 메서드 명확화
  - 테스트 재정렬
- 완료 기준:
  - payment 관련 책임 경계가 명확해짐

### BL-009. Booking / Membership / Attendance 리팩터링

- 배경: 코어 도메인 구현은 되어 있으나 향후 복잡도 증가 가능성 높음
- 해야 할 일:
  - 상태 전이 메서드 정리
  - validator / reader / appender 책임 점검
  - 중복 검증 로직 정리
- 완료 기준:
  - 핵심 도메인 코드의 책임 경계가 더 선명해짐

### BL-010. ErrorCode / validation / 메시지 국제화 준비

- 배경: 국제화 설계 문서 존재, 실제 적용은 아직 전
- 해야 할 일:
  - MessageSource 도입 범위 정의
  - ErrorCode 메시지 분리 전략 정의
  - validation 메시지 국제화 범위 정리
- 완료 기준:
  - 메시지 관리 전략 확정

# 6. 최종 우선순위 요약

현재 시점의 최종 우선순위는 다음과 같다.

1. 실행 환경 및 운영 안정화
2. 결제 후속 운영 항목 마무리
3. 예약/출석 로그 및 테스트 보강
4. 정합성 점검 SQL과 analytics 1차 구현
5. 코어 도메인 리팩터링
6. 관리자 조회/분석 기능 확장
7. OAuth, Feign, Kafka 같은 확장 기술 도입
8. CI/CD, Docker, 관측성 고도화

즉, 현재 백로그의 핵심은 "새 기능 확장"보다 **운영 가능한 백엔드로 다듬는 작업**이다.
