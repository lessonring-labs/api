# 스프린트 아카이브 (Sprint Archive)

이 문서는 LessonRing Backend의 스프린트 기록을 커밋 이력 기준으로 재구성한 아카이브 문서다.

목적:

- 주기별 목표와 실제 진행 내용을 정리한다.
- 무엇이 완료되었고 무엇이 남았는지 맥락을 남긴다.
- 회고 포인트를 축적해 다음 스프린트 계획에 반영한다.

기록 기준:

- 실제 커밋 날짜 기준으로 스프린트를 재구성한다.
- 별도 스프린트 보드가 없는 구간은 커밋 메시지와 문서 상태를 근거로 해석한다.
- 동일 날짜에 여러 커밋이 몰린 경우 하나의 작업 흐름으로 묶는다.

현재 아카이브 기준 범위:

- 2026-03-14 ~ 2026-03-26

# 2. Sprint 1. 코어 도메인 1차 구축

## 기간

- 2026-03-14

## 목표

- LessonRing Backend의 핵심 도메인 골격을 세운다.
- 최소한의 운영 도메인 구조를 코드로 만든다.

## 주요 작업

- 회원, 회원권, 일정, 예약, 출석, 결제 등 핵심 도메인 1차 구현
- 기본 계층 구조 정착
- 도메인 중심 패키지 구조 시작

## 결과

- 프로젝트의 기본 도메인 뼈대가 만들어졌다.
- 이후 고도화할 수 있는 최소 구조가 생겼다.

## 잘된 점

- 초기에 도메인 축을 먼저 세운 방향은 적절했다.
- 이후 결제 및 예약 복잡도를 올릴 수 있는 기반이 생겼다.

## 아쉬운 점

- 이 단계에서는 운영 검증과 테스트 체계가 아직 약했다.
- 코어 흐름은 있었지만 정합성 보강은 이후 단계 과제로 남았다.

## 다음 스프린트로 넘긴 과제

- 코어 도메인 세부 규칙 고도화
- 전체 흐름 검증
- 결제/환불/이벤트 구조 연결

# 3. Sprint 2. 코어 도메인 고도화 및 전체 흐름 연결

## 기간

- 2026-03-16 ~ 2026-03-18

## 목표

- 코어 도메인 2차 고도화
- 전체 도메인 흐름을 연결
- 결제, 회원권, 예약 환불 흐름을 운영 수준으로 끌어올림

## 주요 작업

- 코어 도메인 고도화
- 전체 흐름 안정화 및 검증
- 결제/이용권/예약 환불 흐름 구현
- 이벤트 기반 처리 구조 도입
- Swagger 및 예외 처리 보강

## 결과

- 예약, 결제, 회원권, 환불이 하나의 흐름으로 연결되기 시작했다.
- API 문서화와 예외 처리 체계가 정리되었다.
- 이벤트 기반 후속 처리 방향이 자리를 잡았다.

## 잘된 점

- 단순 CRUD 수준을 넘어서 운영 흐름이 연결되었다.
- 결제 후 회원권 생성, 환불 후 후속 처리 같은 실무 흐름을 다루기 시작했다.

## 아쉬운 점

- 멱등성, 동시성, webhook 안정성은 아직 충분히 다뤄지지 않았다.
- 테스트 범위가 구조 대비 충분히 넓지 않았다.

## 다음 스프린트로 넘긴 과제

- 결제 안정성 강화
- webhook 멱등 처리
- 동시성 제어 보강
- 테스트 확대

# 4. Sprint 3. 결제 안정성 및 확장 설계 1차

## 기간

- 2026-03-24

## 목표

- 결제 흐름의 안정성을 높인다.
- 멱등성과 webhook 처리 구조를 정리한다.
- 앞으로 필요한 로드맵과 지표 문서를 정리한다.

## 주요 작업

- 결제 안정성 강화
- webhook 멱등 처리 구조 확립
- `payment_operation` 구조 추가
- refund 멱등성 2차 적용
- Redis 분산 락 적용
- 동시성 테스트 추가
- 로드맵, 비전 및 지표 문서 정리
- 다국어 설계 문서 추가

## 결과

- 결제는 프로젝트 내에서 가장 정교한 안정성 구조를 가진 영역이 되었다.
- 멱등 처리, 락, webhook 안정화가 핵심 설계 축으로 올라왔다.
- 프로젝트 전체 우선순위와 장기 방향이 문서로 정리되었다.

## 잘된 점

- 운영 사고 리스크가 큰 결제 구간을 먼저 강화한 판단이 적절했다.
- 기술적 보강과 문서 정리를 같은 시기에 진행한 점이 좋았다.

## 아쉬운 점

- analytics, 관리자 조회, 운영 점검은 아직 미구현 상태로 남았다.
- 실행 환경 자동 검증과 배포 체계는 여전히 약했다.

## 다음 스프린트로 넘긴 과제

- approve 2차 안정화
- 교차 충돌 처리
- 테스트 및 QA 문서 체계화
- 아키텍처 문서 정리

- 결제는 이제 기능보다 운영 품질 중심으로 다룰 수 있는 수준까지 올라왔다.

## 아쉬운 점

- 코어 도메인 전체가 결제만큼 테스트와 로그를 갖춘 것은 아니다.
- analytics는 여전히 설계만 있고 구현은 남아 있다.
- 운영 환경 설정, 관측성, 관리자 조회 기능은 다음 단계다.

- `local`, `dev` 실행 검증 자동화

- 핵심 도메인 테스트 확대
- analytics 1차 구현
- 관리자 조회 기능

# 7. 공통 회고

## 잘한 점

- 도메인 중심 구조를 유지했다.
- 결제처럼 리스크가 큰 구간을 우선 강화했다.
- 코드와 문서를 함께 정리하는 흐름이 생겼다.
- 테스트 문서와 QA 문서가 실제로 쌓이기 시작했다.

## 부족한 점

- 코어 도메인 전체 테스트 깊이는 아직 uneven하다.
- 운영 환경 검증 자동화가 없다.
- 로그 표준은 생겼지만 코드 전면 적용은 남아 있다.
- 운영자 관점 조회와 분석 기능이 아직 약하다.

## 앞으로의 개선 방향

- 기능 확장보다 운영 안정화 우선
- 결제 수준의 테스트/로그 기준을 예약/출석/회원권에도 확장
- analytics와 관리자 조회를 1차 구현
- 실행/배포/관측 체계 정비

# 11. Sprint 5 계획안

## 기간 제안

- `2026-03-27 ~ 2026-04-02`

## 배경

현재까지의 커밋 흐름은 `코어 도메인 구축 -> 결제 안정성 강화 -> 테스트/문서화 정리`까지 진행되었다.

다음 단계에서는 새 기능 확장보다 다음 3가지가 우선이다.

- 실행 환경 검증
- 결제 외 코어 도메인 운영 추적 강화
- analytics 1차 구현

## Sprint 5 목표

### 목표 1. 실행 환경을 운영 가능한 수준으로 검증

- `local` 프로필 실행 검증
- `dev` 프로필 설정값 점검
