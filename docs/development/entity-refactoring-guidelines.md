# 엔티티 리팩토링 공통 가이드

이 문서는 `Payment`뿐 아니라 현재 프로젝트의 모든 엔티티에 공통으로 적용할 수 있는 리팩토링 기준을 정리한다.

목표:

- 엔티티 복잡도를 낮춘다.
- 도메인 규칙은 유지하되, 읽기 쉬운 구조로 만든다.
- 엔티티, enum, 서비스, 값 객체의 책임 경계를 명확히 한다.

---

# 기본 원칙

엔티티는 다음 책임만 우선적으로 가진다.

- 상태 보관
- 핵심 불변식 유지
- 자기 자신에 대한 상태 변경

엔티티가 과도하게 가지면 안 되는 책임은 다음과 같다.

- 여러 엔티티를 동시에 조정하는 흐름
- 외부 시스템 응답 해석
- 화면/API 전용 포맷팅
- 긴 정책 판단 로직
- 동적 쿼리 구성

---

# 복잡한 엔티티의 대표 신호

아래 항목 중 여러 개가 보이면 구조 분리가 필요하다.

- public 메서드 하나가 너무 길다.
- 상태 변경 전에 검증이 과도하게 많다.
- 문자열 기반 예외 메시지와 조건문이 반복된다.
- 외부 연동 결과를 직접 해석한다.
- 한 메서드의 파라미터 수가 많다.
- 한 엔티티에서 여러 비즈니스 시나리오를 다 처리한다.

---

# 권장 구조

## 1. 필드 구간과 메서드 구간을 분리

엔티티는 읽는 순서를 통일하는 것이 좋다.

권장 순서:

1. 상수
2. 필드
3. 생성자
4. 정적 팩토리 메서드
5. public 상태 변경 메서드
6. private 검증 메서드
7. private 값 반영 메서드

이 구조만 맞춰도 가독성이 많이 올라간다.

---

## 2. public 메서드는 시나리오만 보여주기

public 메서드는 "무엇을 하는가"가 보여야 한다.

좋은 예:

```java
public void cancel(LocalDateTime canceledAt) {
    validateCancelable();
    applyCancel(canceledAt);
}
```

나쁜 예:

```java
public void cancel(LocalDateTime canceledAt) {
    if (...) { ... }
    if (...) { ... }
    if (...) { ... }
    this.status = ...
    this.canceledAt = ...
}
```

즉, public 메서드는 시나리오를 보여주고 세부 구현은 private 메서드로 숨긴다.

---

## 3. 상태 전이 규칙은 enum으로 이동

상태성 엔티티는 enum을 적극 활용하는 것이 좋다.

예:

- `BookingStatus.canCancel()`
- `PaymentStatus.canApprove()`
- `MembershipStatus.isUsable()`

효과:

- 상태 규칙이 enum 한 곳에 모인다.
- 엔티티 조건문이 짧아진다.
- 테스트가 쉬워진다.

---

## 4. 검증 메서드와 상태 변경 메서드를 분리

긴 메서드는 대부분 다음 두 단계로 쪼갤 수 있다.

- `validate...`
- `apply...`

예:

```java
public void use() {
    validateUsable();
    applyUse();
}
```

이 패턴은 거의 모든 엔티티에 공통 적용 가능하다.

---

## 5. 파라미터가 많으면 입력 객체로 묶기

메서드 인자가 많으면 의미가 흐려진다.

후보:

- 승인 결과
- 실패 정보
- 스냅샷 정보
- 변경 요청 정보

예:

- `PaymentApproveCommand`
- `BookingCancelPolicyInput`
- `MembershipSnapshot`

효과:

- 메서드 시그니처 단순화
- 의미 단위가 분명해짐
- 테스트 데이터 준비가 쉬워짐

---

## 6. 값 묶음은 값 객체로 분리

아래처럼 항상 같이 움직이는 필드는 값 객체 후보이다.

- 기간
- 금액
- PG 정보
- 스냅샷 정보
- 연락처 정보

방법:

- `@Embeddable`
- 일반 도메인 값 객체
- 입력 전용 command 객체

---

## 7. 외부 시스템 개념은 엔티티에 최소만 남기기

엔티티가 외부 시스템 세부 구현을 과하게 알면 복잡도가 커진다.

예:

- PG 응답 해석
- Redis 락 처리
- webhook 중복 판정
- API 응답 포맷

이런 책임은 서비스 또는 어댑터 계층으로 이동한다.

---

## 8. 애그리거트 루트와 단순 엔티티를 구분

모든 엔티티가 rich domain model일 필요는 없다.

상대적으로 규칙이 많은 엔티티:

- `Payment`
- `Booking`
- `Membership`

상대적으로 단순하게 유지해도 되는 엔티티:

- `Notification`
- `RefreshToken`
- `PaymentWebhookLog`

즉, 모든 엔티티에 같은 수준의 메서드를 넣지 않는다.

---

# 전체 엔티티에 적용할 실무 규칙

권장 기준:

- public 메서드는 가능하면 5~15줄 내외
- 조건문이 3개를 넘으면 분리 검토
- 파라미터가 5개를 넘으면 입력 객체 검토
- 외부 시스템 값이 많이 보이면 서비스 분리 검토
- 같은 예외 메시지 패턴이 반복되면 검증 메서드 정리

---

# 적용 우선순위

전체 엔티티를 한 번에 손대기보다 우선순위를 두는 것이 좋다.

1. 상태 전이와 조건문이 많은 엔티티
2. 파라미터가 많은 메서드를 가진 엔티티
3. 서비스 로직이 엔티티로 과하게 들어온 엔티티
4. 값 묶음이 많은 엔티티

현재 프로젝트에서는 다음 순서가 현실적이다.

1. `Payment`
2. `Booking`
3. `Membership`
4. `Attendance`
5. 나머지 단순 엔티티

---

# 리팩토링할 때 피해야 할 것

- 무조건 엔티티를 빈 껍데기로 만드는 것
- 상태 전이 규칙까지 전부 서비스로 빼는 것
- 값 객체를 과하게 잘게 쪼개는 것
- 한 번에 모든 엔티티를 같은 방식으로 바꾸는 것

핵심은 균형이다.

- 엔티티는 자기 상태를 안전하게 바꿀 수 있어야 한다.
- 하지만 정책, 오케스트레이션, 외부 연동 세부사항까지 다 들고 있으면 안 된다.

---

# 결론

엔티티 리팩토링의 목표는 메서드 수를 줄이는 것이 아니라 책임을 읽기 좋게 재배치하는 것이다.

현재 프로젝트에서는 다음 원칙이 가장 중요하다.

- 상태 전이는 엔티티 안에 남긴다.
- 상태 가능 여부 판단은 enum이나 검증 메서드로 이동한다.
- 큰 시나리오는 서비스로 분리한다.
- 값 묶음은 값 객체로 정리한다.

이 기준으로 정리하면 `Payment`뿐 아니라 다른 엔티티도 훨씬 읽기 쉬워진다.
