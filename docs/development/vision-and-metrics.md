# 비전 및 지표

이 문서는 현재 LessonRing Backend 프로젝트의 비전과 이를 측정하기 위한 핵심 지표를 정리한다.

기준 시점:

- 2026-03-24

목표:

- 프로젝트가 어디를 향하는지 명확히 한다.
- 기능 개발과 운영 안정화의 우선순위를 수치로 연결한다.
- 사업 지표와 기술 지표를 혼동하지 않고 분리해서 관리한다.

---

# 1. 프로젝트 비전

LessonRing Backend의 비전은 다음과 같다.

- 레슨 스튜디오 운영에 필요한 핵심 업무를 하나의 일관된 흐름으로 관리하는 백엔드
- 예약, 출석, 이용권, 결제 흐름이 서로 충돌하지 않고 안정적으로 이어지는 시스템
- 운영자가 실제 현장에서 신뢰할 수 있는 데이터와 상태를 제공하는 시스템

즉, 단순 기능 수를 늘리는 것이 아니라:

- 상태가 맞게 흘러가고
- 결제와 예약이 안전하게 처리되고
- 운영 중 문제를 추적할 수 있는 백엔드

를 만드는 것이 핵심이다.

---

# 2. 제품 관점 목표

현재 프로젝트의 제품 관점 목표는 아래 세 가지로 정리할 수 있다.

## 목표 1. 운영 핵심 흐름 통합

운영자가 다음 흐름을 한 시스템 안에서 이어서 관리할 수 있어야 한다.

- 회원 등록
- 이용권 생성 및 상태 관리
- 수업 등록 및 조회
- 예약 생성 및 취소
- 출석 처리
- 결제 승인 및 환불

## 목표 2. 결제/예약 신뢰성 확보

가장 중요한 운영 리스크는 결제와 예약의 불일치다.

따라서 다음이 중요하다.

- 중복 예약 방지
- 중복 결제 처리 방지
- webhook 중복 수신 안전 처리
- 환불 후 이용권/예약 상태 정합성 유지

## 목표 3. 운영 추적 가능성 확보

문제가 생겼을 때 다음이 가능해야 한다.

- 어떤 회원이 어떤 상태인지 확인
- 어떤 결제가 어떤 흐름으로 처리됐는지 추적
- 환불과 webhook 처리 이력을 확인
- 알림, 출석, 이용권 상태를 함께 해석

---

# 3. 단계별 비전

## 단기 비전

- 현재 구현된 핵심 도메인을 안정적으로 실행 가능한 상태로 만든다.
- 운영자가 결제/예약/이용권 흐름을 신뢰할 수 있게 만든다.

## 중기 비전

- 관리 화면과 통계/조회 기능을 강화한다.
- 다국어, 알림, 외부 연동 기반을 단계적으로 확장한다.

## 장기 비전

- 이벤트 기반 확장, 외부 플랫폼 연동, 배포/운영 자동화까지 포함한 서비스 운영 플랫폼으로 발전시킨다.

---

# 4. 핵심 성공 지표 구조

현재 단계에서는 지표를 4개 그룹으로 나눠 관리하는 것이 적절하다.

1. 제품/운영 지표
2. 결제/정합성 지표
3. API/시스템 품질 지표
4. 개발 생산성 지표

---

# 5. 제품 / 운영 지표

현재 프로젝트가 실제 운영 업무를 얼마나 잘 지원하는지 보는 지표다.

## 회원 운영 지표

- 총 회원 수
- 활성 회원 수
- 최근 30일 신규 회원 수

의미:

- 실제 운영 데이터가 쌓이고 있는지 확인

## 이용권 운영 지표

- 활성 이용권 수
- 만료 예정 이용권 수
- 사용 완료 이용권 수
- 환불된 이용권 수

의미:

- 회원 운영과 매출 흐름의 핵심 상태 확인

## 예약 운영 지표

- 총 예약 수
- 예약 완료 수
- 취소 수
- 노쇼 수
- 출석 완료 수

의미:

- 실제 수업 운영 상태 파악

---

# 6. 결제 / 정합성 지표

현재 프로젝트에서 가장 중요한 기술-운영 혼합 지표다.

## 결제 성공 지표

- 결제 생성 수
- 결제 완료 수
- 결제 실패 수
- 결제 취소 수

핵심 비율:

- 결제 완료율 = 완료 수 / 결제 생성 수
- 결제 실패율 = 실패 수 / 결제 생성 수

## 멱등 / 중복 방지 지표

- 중복 승인 요청 수
- 중복 webhook 수신 수
- idempotency key 재사용 감지 수
- payment_operation 중복 차단 수

의미:

- 현재 프로젝트의 결제 안정성 구조가 실제로 얼마나 의미 있게 작동하는지 측정

## 정합성 지표

- 결제 완료 후 이용권 미생성 건수
- 환불 완료 후 이용권 환불 미반영 건수
- 환불 완료 후 예약 취소 미반영 건수
- 출석 처리 후 이용권 차감 불일치 건수

이 지표는 특히 중요하다.

이유:

- 운영자가 실제로 겪는 치명적인 문제는 대부분 "처리는 됐는데 상태가 안 맞는 경우"이기 때문이다.

---

# 7. API / 시스템 품질 지표

## API 안정성 지표

- API 성공률
- 4xx 비율
- 5xx 비율
- 평균 응답 시간
- P95 응답 시간

우선 관찰 대상 API:

- 회원 조회/등록
- 예약 생성/취소
- 출석 처리
- 결제 생성/승인/환불

## DB / Migration 지표

- migration 실패 건수
- 프로필별 부팅 성공률
- baseline 기준 신규 환경 초기화 성공 여부

## Redis / 락 지표

- 락 획득 실패 수
- 락 대기 시간
- 락 관련 예외 건수

의미:

- 예약/결제 동시성 문제가 있는지 확인

---

# 8. 개발 생산성 지표

## 테스트 지표

- 핵심 서비스 테스트 커버 여부
- 결제/예약/이용권 테스트 시나리오 수
- 회귀 버그 재발 건수

## 변경 안정성 지표

- 릴리스 후 긴급 수정 건수
- migration 재작업 건수
- 문서와 실제 코드 불일치 발견 건수

## 문서화 지표

- 도메인 문서 최신화율
- migration 문서 최신화율
- 설정 문서 최신화율

현재 프로젝트는 문서 기반 정리가 많이 진행됐으므로, 이 지표를 유지하는 것이 중요하다.

---

# 9. 현재 단계에서 가장 중요한 KPI

지금 시점에서는 모든 지표를 동일한 무게로 보지 않는 것이 좋다.

우선순위가 높은 KPI는 아래다.

## 1순위

- 결제 완료 후 상태 정합성 오류 건수
- 환불 후 상태 정합성 오류 건수
- 중복 webhook 처리 차단 건수
- 예약 중복 생성 오류 건수

## 2순위

- 핵심 API 5xx 비율
- 결제/예약 API 평균 응답 시간
- `local`, `dev` 부팅 성공 여부

## 3순위

- 활성 회원 수
- 활성 이용권 수
- 예약 완료/취소/노쇼 비율

---

# 10. 추천 목표값

아직 초기 운영 단계이므로, 절대 수치보다 "이상 징후를 빠르게 발견하는 기준"이 중요하다.

초기 목표 예시:

- 결제 완료 후 이용권 미생성 건수: `0`
- 환불 후 이용권 환불 미반영 건수: `0`
- 중복 예약 허용 건수: `0`
- webhook 중복 수신으로 인한 중복 처리 건수: `0`
- 핵심 API 5xx 비율: `1% 미만`
- `local`, `dev` 프로필 부팅 성공률: `100%`

이 단계에서는 "작게라도 정확히 보는 지표"가 중요하다.

- 알림 임계치 설정
- 배포/운영 지표 자동화

# 13. 결론

현재 프로젝트에서 비전과 지표의 핵심은 다음으로 요약할 수 있다.

- 비전: 레슨 스튜디오 운영 핵심 흐름을 안정적으로 연결하는 백엔드
- 제품 목표: 회원, 이용권, 예약, 출석, 결제를 하나의 신뢰 가능한 흐름으로 묶는 것
- 핵심 지표: 결제와 예약의 정합성, 운영 안정성, API 품질

즉, 지금 단계에서 가장 중요한 것은

- 기능 수 확대보다
- 상태 정합성과 운영 신뢰도 측정

을 먼저 지표화하는 것이다.
