# 팀 구성 및 R&R (Team & Roles)

이 문서는 2026-03 기준 LessonRing Backend 코드베이스를 분석한 결과를 바탕으로, **3인 팀 기준 프로젝트 참여 인원과 담당 업무를 정의**하기 위한 문서다.

목표는 단순 기능 분배가 아니라 다음을 만족하는 역할 구성을 만드는 것이다.

- 운영 리스크가 큰 영역의 책임자를 명확히 둔다.
- 도메인 간 결합도가 높은 지점을 누가 조정할지 정한다.
- 개발, 테스트, 배포, 장애 대응의 공백을 줄인다.
- 향후 인원 증원 시 역할 확장이 쉬운 구조를 만든다.

---

# 1. 프로젝트 분석 요약

## 1.1 프로젝트 성격

LessonRing Backend는 단순 CRUD 백엔드가 아니다. 현재 코드 기준으로 다음 특성이 뚜렷하다.

- 스튜디오 운영형 백엔드
- 도메인 중심 패키지 구조
- 모듈형 모놀리식 구조
- 예약, 출석, 회원권, 결제, 알림이 강하게 연결된 운영 시스템
- 결제 및 예약 구간에 동시성 제어와 멱등 처리 요구가 존재

즉, 화면이나 API 개수보다 **운영 흐름의 정확성**이 더 중요한 프로젝트다.

## 1.2 현재 구현된 핵심 도메인

현재 주요 구현 범위는 다음과 같다.

- 인증: `auth`, JWT, refresh token
- 운영 기준정보: `studio`, `instructor`
- 회원 관리: `member`
- 상품/이용권: `membership`
- 수업 일정: `schedule`
- 예약/취소/no-show: `booking`
- 출석 처리: `attendance`
- 결제/환불/웹훅/멱등성: `payment`
- 이벤트 기반 알림 생성: `notification`
- 조회 확장 기반: `analytics`

## 1.3 기술 및 운영 특성

현재 코드와 문서 기준 기술적 난이도가 높은 축은 다음과 같다.

- `PostgreSQL + JPA + Querydsl + Flyway` 기반 데이터 모델 관리
- `Redis + Redisson` 기반 예약/결제 동시성 제어
- `Toss Payments + Webhook` 기반 외부 결제 연동
- `Spring Security + JWT` 기반 인증/인가
- 도메인 이벤트 기반 알림 처리

## 1.4 역할 분리 시 고려해야 할 핵심 포인트

3명 기준 역할을 나눌 때 가장 중요한 기준은 다음이다.

- 예약/출석/회원권은 운영 코어 도메인이라 한 축으로 묶는 것이 맞다.
- 결제/환불/웹훅은 외부 연동과 장애 리스크가 커서 별도 오너가 필요하다.
- 인증, 공통 인프라, DB 마이그레이션, 테스트 기준, 배포 기준은 누군가 전담하지 않으면 팀 전체 품질이 빠르게 흔들린다.

따라서 3인 구성은 다음 3축이 가장 현실적이다.

- 운영 코어 도메인
- 결제 및 외부 연동
- 플랫폼 및 품질 체계

---

# 2. 3인 팀 구성 제안

## 2.1 권장 팀 구조

### 1) PO 겸 백엔드 리드

- 역할명: `Backend Lead / Core Domain Owner`
- 주 책임: 예약, 출석, 회원권, 일정 등 운영 핵심 흐름 총괄

### 2) 결제/통합 백엔드 엔지니어

- 역할명: `Payment & Integration Engineer`
- 주 책임: 결제 승인, 환불, PG 연동, webhook, 외부 시스템 연계

### 3) 플랫폼/품질 백엔드 엔지니어

- 역할명: `Platform & Quality Engineer`
- 주 책임: 인증, 공통 모듈, DB/배포/테스트 체계, 운영 안정성

이 구조는 "기능 3등분"이 아니라 "리스크 기준 3등분"이다. 현재 프로젝트에서는 이 방식이 가장 충돌이 적고 책임 경계가 명확하다.

# 3. 역할별 R&R

## 3.1 Backend Lead / Core Domain Owner

### 담당 범위

- `booking`
- `attendance`
- `membership`
- `schedule`
- `member`
- `studio`
- `instructor`

### 핵심 책임

- 운영 핵심 도메인 정책 정의
- 예약 생성/취소/no-show 정책 관리
- 출석 처리와 회원권 차감/복구 정책 관리
- 스케줄 정원, 예약 가능 조건, 운영 상태값 설계
- 도메인 간 정책 충돌 조정
- API 우선순위 및 기능 로드맵 관리

### 산출물 책임

- 핵심 도메인 요구사항 정리
- 도메인 규칙 문서 최신화
- 운영 정책 변경 시 영향도 분석
- 핵심 유스케이스 테스트 시나리오 정의

### 의사결정 권한

- 예약/출석/회원권 규칙 변경 승인
- 도메인 모델 구조 변경 방향 결정
- 정책성 예외 처리 기준 확정

### KPI 예시

- 예약/출석 관련 운영 이슈 건수
- 회원권 차감 오류 건수
- 핵심 도메인 테스트 커버리지 및 회귀 이슈 수

## 3.2 Payment & Integration Engineer

### 담당 범위

- `payment`
- `integration`
- `common.event` 일부 연동 지점
- 외부 PG 및 webhook 관련 설정

### 핵심 책임

- 결제 생성, 승인, 취소, 환불 로직 개발 및 유지보수
- PG 연동 안정성 확보
- webhook 중복 처리, 멱등성, 상태 정합성 보장
- 외부 API 연동 코드 표준화
- 결제 장애 대응 및 정산 이슈 분석 지원
- 향후 Kakao OAuth, Feign, Kafka 도입 시 연동 설계 주도

### 산출물 책임

- 결제 흐름 문서
- 외부 연동 장애 대응 가이드
- PG sandbox 및 운영 시나리오 테스트 케이스
- 멱등 처리 및 webhook 처리 정책 문서

### 의사결정 권한

- 결제 상태 전이 방식
- webhook 처리 재시도/무시 기준
- 외부 연동 인터페이스 설계 기준

### KPI 예시

- 결제 승인/환불 실패율
- webhook 중복 처리 오류 건수
- 결제 장애 평균 복구 시간

## 3.3 Platform & Quality Engineer

### 담당 범위

- `auth`
- `common`
- `src/main/resources/application*.yml`
- `db/migration`
- 테스트 기반, 보안 설정, 운영 환경 표준
- 빌드/배포/문서 체계

### 핵심 책임

- JWT, Spring Security, 인증/인가 정책 유지
- 공통 예외, 응답 규격, 설정 관리 체계 정비
- Flyway migration 정책 관리
- 테스트 전략 수립 및 회귀 자동화 기반 강화
- 환경 분리, 실행 설정, 배포 표준화
- 관측성, 로그, 헬스체크, 장애 대응 기준 정비

### 산출물 책임

- 개발/테스트/배포 가이드
- 보안 및 설정 표준 문서
- DB 마이그레이션 기준서
- 테스트 실행 기준 및 품질 체크리스트

### 의사결정 권한

- 공통 모듈 구조와 의존 규칙
- 인증/보안 설정 정책
- 마이그레이션 반영 절차
- 품질 게이트와 머지 기준

### KPI 예시

- 배포 실패율
- 인증/권한 관련 장애 건수
- 테스트 누락으로 인한 회귀 건수
- 환경 설정 오류 건수

# 4. 역할 간 협업 경계

## 4.1 Backend Lead와 Payment Engineer의 경계

다음은 공동 설계가 필요하다.

- 결제 완료 후 회원권 생성
- 환불 시 미래 예약 취소 정책
- no-show, 출석, 환불 정책 간 상호 영향

정책 오너는 `Backend Lead`가 맡고, 구현 오너는 결제 흐름이면 `Payment Engineer`, 운영 코어 흐름이면 `Backend Lead`가 맡는 것이 적절하다.

## 4.2 Backend Lead와 Platform Engineer의 경계

다음은 공동 관리가 필요하다.

- 도메인 서비스의 트랜잭션 전략
- Redis 락 적용 기준
- 테스트 우선순위와 회귀 범위

도메인 규칙은 `Backend Lead`, 공통 기술 기준은 `Platform Engineer`가 책임진다.

## 4.3 Payment Engineer와 Platform Engineer의 경계

다음은 공동 관리가 필요하다.

- webhook 보안 검증
- 외부 API 인증 및 설정값 관리
- 장애 로그, 추적, 재처리 운영 기준

결제 연동 정책은 `Payment Engineer`, 보안/설정/운영 표준은 `Platform Engineer`가 맡는다.

---

# 5. 공통 책임

3명 모두에게 공통으로 필요한 책임은 다음과 같다.

- 코드리뷰 참여
- 테스트 코드 작성
- 문서 최신화
- 장애 재현 및 원인 공유
- API 변경 시 계약 영향 확인

단, 모든 사람이 공통 책임을 가진다고 해서 오너십이 없어지면 안 된다. 각 영역은 반드시 **최종 책임자 1명**이 있어야 한다.

# 6. 권장 의사결정 구조

## 6.1 최종 의사결정자

- 도메인 정책: `Backend Lead`
- 결제/외부 연동 정책: `Payment Engineer`
- 공통 기술 기준/배포/보안: `Platform Engineer`

## 6.2 충돌 시 정렬 기준

의견 충돌이 생기면 다음 우선순위로 결정한다.

1. 운영 정합성
2. 금전/데이터 무결성
3. 장애 복구 용이성
4. 개발 생산성
5. 코드 미려함

이 프로젝트는 예약과 결제를 다루므로, 생산성보다 정합성과 복구 가능성이 우선이다.

# 7. 실제 업무 분장 예시

## 7.1 신규 기능 개발

예: "결제 완료 시 자동 알림 + 예약 가능 회원권 자동 활성화"

- Backend Lead: 회원권 활성화 정책 정의, 예약 가능 조건 영향 분석
- Payment Engineer: 결제 완료 이벤트 반영, PG 상태 연동 처리
- Platform Engineer: 이벤트 흐름 테스트, 설정값, 운영 로그, 예외 응답 검증

## 7.2 장애 대응

예: "webhook 중복 호출로 결제 상태 불일치 발생"

- Payment Engineer: 원인 분석 및 재처리 로직 점검
- Platform Engineer: 로그/추적/재현 환경 확보, 배포 안정성 검토
- Backend Lead: 회원권/예약 데이터 영향도 확인 및 운영 보정 판단

## 7.3 구조 개선

예: "Payment, Booking 엔티티 책임 재정리"

- Backend Lead: 도메인 규칙 관점 리뷰
- Payment Engineer: 결제 도메인 리팩터링 수행
- Platform Engineer: 테스트 보강, 트랜잭션/의존 규칙 검토

# 8. 팀장 또는 PM 관점 운영 포인트

3명 팀에서는 역할 분리보다 더 중요한 것이 우선순위 통제다. 현재 코드 기준으로 추천 운영 우선순위는 다음과 같다.

1. 예약/결제/회원권 정합성 유지
2. 회귀 테스트 확대
3. 배포 및 운영 기준 정리
4. 분석/조회 기능 확장
5. 신규 외부 연동 도입

즉, 현 단계에서 가장 먼저 챙겨야 할 인력 투입처는 신규 기능 확장보다 **코어 흐름 안정화**다.

# 9. 결론

LessonRing Backend를 3명 기준으로 운영한다면 다음 구성이 가장 적절하다.

- `1명`: 운영 코어 도메인 오너
- `1명`: 결제 및 외부 연동 오너
- `1명`: 플랫폼, 인증, 품질, 배포 오너

현재 프로젝트는 예약, 회원권, 출석, 결제가 서로 강하게 연결되어 있기 때문에, 단순히 API 개수로 역할을 나누면 경계가 흐려지고 장애 대응이 어려워진다.

반대로 위와 같이 **코어 도메인 / 결제 연동 / 플랫폼 품질** 기준으로 분리하면 다음 장점이 있다.

- 책임 경계가 명확하다.
- 운영 이슈 대응이 빠르다.
- 결제와 예약의 정합성을 유지하기 쉽다.
- 인원 확장 시 도메인별 세분화가 쉽다.

현재 LessonRing Backend의 코드 상태와 로드맵을 기준으로 보면, 이 3인 R&R 구조가 가장 현실적이고 유지보수 친화적인 팀 구성안이다.
