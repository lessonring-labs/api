# 결제 환불 통합 테스트 설계서

## 1. 대상 정보

- 테스트 파일: [PaymentServiceIntegrationTest.java](/C:/wms/api/src/test/java/com/lessonring/api/common/security/payment/application/PaymentServiceIntegrationTest.java)
- 대상 계층: 결제 환불 애플리케이션 서비스
- 테스트 유형: 통합 테스트

## 2. 문서 목적

환불 도메인의 핵심 정책이 실제 저장소, 이벤트, 이용권, 예약 상태 변경과 함께 올바르게 동작하는지 통합 수준에서 검증한다.

## 3. 검증 범위

- 환불 가능 상태 검증
- 이미 환불된 이용권과의 관계 검증
- 환불 동시성 및 멱등성
- 환불 성공 후 결제 상태 변경
- 이용권 상태 변경
- 미래 예약 자동 취소
- 환불 금액 계산
- 환불 이벤트 발행 및 알림 후속 처리

## 4. 비검증 범위

- 프론트엔드 환불 UI
- 실제 외부 PG 네트워크 호출

## 5. 사전 조건

- 테스트 데이터용 회원, 이용권, 결제, 예약, 스케줄 생성이 가능해야 한다.
- 외부 PG는 [TestExternalMockConfig.java](/C:/wms/api/src/test/java/com/lessonring/api/support/TestExternalMockConfig.java)로 대체되어야 한다.
- 트랜잭션과 락 정책이 테스트 환경에서 작동해야 한다.

## 6. 핵심 업무 규칙

- `COMPLETED` 상태 결제만 환불할 수 있다.
- 이미 `REFUNDED` 상태인 이용권이 연결된 결제는 재환불할 수 없다.
- 환불 시 미래 `RESERVED` 예약은 취소되어야 한다.
- COUNT / PERIOD 이용권별 환불 계산 규칙이 달라질 수 있다.

## 7. 상세 테스트 케이스

| ID | 시나리오 | 입력 조건 | 수행 절차 | 기대 결과 |
|-----|-----|-----|-----|-----|
| PAY-REF-INT-001 | 완료되지 않은 결제 환불 차단 | `READY` 또는 `FAILED` 결제 | 환불 호출 | 비즈니스 예외 발생 |
| PAY-REF-INT-002 | 이미 환불된 이용권 연결 결제 재환불 차단 | 이용권 상태 `REFUNDED` | 환불 호출 | 비즈니스 예외 발생 |
| PAY-REF-INT-003 | 서로 다른 `idempotencyKey` 동시 환불 | 동일 paymentId, 서로 다른 키 | 2개 스레드 동시 환불 | 1건만 성공, 1건 실패 |
| PAY-REF-INT-004 | 동일 `idempotencyKey` 동시 환불 | 동일 paymentId, 동일 키 | 2개 스레드 동시 환불 | 실제 처리 1회, 재처리 방지 |
| PAY-REF-INT-005 | 정상 환불 성공 | 완료된 결제와 유효 이용권 | 환불 호출 | 결제 `CANCELED`, 이용권 `REFUNDED` |
| PAY-REF-INT-006 | 횟수형 이용권 환불 금액 계산 | COUNT 이용권 | 환불 호출 | 잔여 횟수 기반 금액 계산 일치 |
| PAY-REF-INT-007 | 기간형 이용권 환불 금액 계산 | PERIOD 이용권 | 환불 호출 | 잔여 기간 기반 금액 계산 일치 |
| PAY-REF-INT-008 | 미래 예약 자동 취소 | 미래 `RESERVED` 예약 존재 | 환불 호출 | 미래 예약 취소, 과거 출석 예약 유지 |
| PAY-REF-INT-009 | 환불 이벤트 발행 | 정상 환불 성공 | 환불 호출 후 후속 확인 | 결제 취소 이벤트와 알림 처리 수행 |
| PAY-REF-INT-010 | 잔여 횟수 없는 COUNT 이용권 환불 차단 | 사용 완료 상태 | 환불 호출 | 비즈니스 예외 발생 |
| PAY-REF-INT-011 | 만료된 PERIOD 이용권 환불 차단 | 만료 상태 | 환불 호출 | 비즈니스 예외 발생 |

## 8. 합격 기준

- 결제, 이용권, 예약 상태가 정책대로 함께 변경되어야 한다.
- 동시 요청 환경에서도 중복 환불이 발생하면 실패다.
- 환불 금액 계산 결과가 정책과 다르면 실패다.

## 9. 운영 관점 중요도

높음. 금전 정산, CS, 이용권 복구 문제와 직접 연결되는 핵심 회귀 테스트다.
