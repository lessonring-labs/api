# 기술 결정 기록 (ADR)

이 문서는 LessonRing Backend의 주요 기술적 선택에 대한 이유와 맥락을 기록하는 ADR(Architecture Decision Record) 문서다.

목적:

- 왜 이 기술과 구조를 선택했는지 기록한다.
- 현재 코드 상태와 설계 의도를 함께 남긴다.
- 향후 기술 변경 시 비교 기준을 제공한다.
- 새 팀원이 빠르게 의사결정 맥락을 이해할 수 있게 한다.

상태 정의:

- `Accepted`: 현재 채택되어 사용 중
- `Pending`: 도입 후보지만 아직 확정되지 않음
- `Superseded`: 다른 결정으로 대체됨

---

# ADR-001. 모듈형 모놀리식 구조 채택

- 상태: `Accepted`

## 맥락

회원, 회원권, 일정, 예약, 출석, 결제가 강하게 연결된 운영형 시스템이다.

## 결정

전체 구조는 `모듈형 모놀리식`으로 간다.

## 이유

- 도메인 간 트랜잭션과 정합성 관리가 쉽다.
- 초기 단계에서 분산 시스템 복잡도를 피할 수 있다.
- 도메인 경계를 유지하면서도 개발 속도를 확보할 수 있다.

---

# ADR-002. 도메인 중심 패키지 구조 채택

- 상태: `Accepted`

## 결정

도메인별 패키지 아래에 `api`, `application`, `domain`, `infrastructure` 계층을 둔다.

## 이유

- 도메인 경계가 코드에서 명확해진다.
- 변경 범위를 좁히기 쉽다.
- 온보딩과 유지보수에 유리하다.

---

# ADR-003. Java 21 + Spring Boot 3 채택

- 상태: `Accepted`

## 결정

런타임은 `Java 21`, 프레임워크는 `Spring Boot 3.5.9`를 사용한다.

## 이유

- 장기 지원 버전 기반으로 안정적이다.
- 보안, 데이터 접근, 검증, 설정 관리가 통합된다.
- 운영형 REST API 백엔드에 적합하다.

---

# ADR-004. PostgreSQL + JPA/Hibernate 채택

- 상태: `Accepted`

## 결정

주 저장소는 `PostgreSQL`, ORM은 `Spring Data JPA / Hibernate`를 사용한다.

## 이유

- 예약, 결제, 회원권은 강한 트랜잭션 보장이 필요하다.
- 관계형 모델이 도메인 구조와 잘 맞는다.
- Spring 생태계와 통합이 안정적이다.

---

# ADR-005. Querydsl로 복잡한 조회 분리

- 상태: `Accepted`

## 결정

복잡한 조회는 `Querydsl` 기반 Query Repository로 분리한다.

## 이유

- 동적 조건 조회에 유리하다.
- JPQL 문자열보다 유지보수성이 좋다.
- 관리자 조회와 analytics 확장에 적합하다.

---

# ADR-006. Flyway로 DB 마이그레이션 관리

- 상태: `Accepted`

## 결정

DB 스키마 변경은 `Flyway` 마이그레이션으로 관리한다.

## 이유

- 스키마 이력을 명시적으로 관리할 수 있다.
- 환경 간 차이를 줄일 수 있다.
- 민감한 테이블 변경 추적이 쉽다.

---

# ADR-007. JWT 기반 Stateless 인증 채택

- 상태: `Accepted`

## 결정

인증은 `Spring Security + JWT` 기반 stateless 방식으로 간다.

## 이유

- 웹/모바일/API 클라이언트 구조에 적합하다.
- 서버 세션 상태 관리 부담이 줄어든다.
- API 중심 아키텍처와 잘 맞는다.

---

# ADR-008. Redis + Redisson으로 예약/결제 동시성 제어

- 상태: `Accepted`

## 결정

예약과 결제의 충돌 구간은 `Redis + Redisson` 분산 락으로 제어한다.

## 이유

- 예약 정원 초과와 결제 중복은 운영 사고로 직결된다.
- 애플리케이션 레벨에서 자원 단위 직렬화가 필요하다.
- DB 트랜잭션과 함께 사용하기 좋다.

---

# ADR-009. Toss Payments + Webhook 결제 연동 채택

- 상태: `Accepted`

## 결정

결제 연동은 `Toss Payments`를 사용하고, 상태 동기화는 `Webhook`을 병행한다.

## 이유

- 승인, 취소, 환불 API가 명확하다.
- webhook을 통한 비동기 상태 보정이 가능하다.
- 결제 후 회원권 생성, 환불 후 예약 취소 흐름과 잘 맞는다.

---

# ADR-010. 결제 멱등 처리 테이블 별도 관리

- 상태: `Accepted`

## 결정

결제 승인과 환불 멱등성은 `payment_operation` 테이블로 관리한다.

## 이유

- 중복 요청 추적과 차단이 가능하다.
- 성공/실패 이력 복원이 가능하다.
- 외부 연동 장애 분석이 쉬워진다.

---

# ADR-011. 도메인 이벤트로 후속 처리 분리

- 상태: `Accepted`

## 결정

핵심 유스케이스 이후 알림 등 후속 처리는 도메인 이벤트 기반으로 분리한다.

## 이유

- 도메인 간 직접 결합을 낮출 수 있다.
- 알림, 분석, 외부 연동 확장에 유리하다.
- 후속 처리 책임을 논리적으로 분리할 수 있다.

---

# ADR-012. OpenAPI / Swagger UI 문서 자동화 채택

- 상태: `Accepted`

## 결정

API 문서는 `Springdoc OpenAPI + Swagger UI`로 자동화한다.

## 이유

- 클라이언트와의 계약 확인이 쉽다.
- 테스트와 연동 생산성이 높다.
- 운영 API를 빠르게 탐색할 수 있다.

---

# ADR-013. Kafka, Feign, MapStruct는 도입 후보로 유지

- 상태: `Pending`

## 결정

Kafka, Feign, MapStruct는 현재 채택 완료가 아니라 후보 기술로 유지한다.

## 이유

- 코드 흔적은 있으나 실제 의존성과 운영 기준이 확정되지 않았다.
- 현재 제품 단계에서는 필수 운영 가치보다 준비 코드 성격이 강하다.
- premature adoption을 피해야 한다.

---

# ADR 운영 규칙

1. 중요한 기술 선택이 생기면 새 ADR 번호를 추가한다.
2. 기존 선택을 뒤집을 때는 새 ADR로 대체 관계를 기록한다.
3. 구현이 되지 않은 선택은 `Accepted`로 기록하지 않는다.
4. 코드와 문서가 어긋나면 코드 기준으로 갱신한다.
