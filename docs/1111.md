# Backend4 - Approve / Refund / Webhook 교차 충돌 테스트 케이스 목록

## 1. 목적

approve / refund / webhook 는 각각 단독 테스트만으로 충분하지 않다.  
같은 payment에 대해 동시에 진입할 때도 상태 정합성이 유지되어야 한다.

이 문서는 아래를 검증하기 위한 교차 충돌 테스트 케이스 목록이다.

- payment 상태 정합성
- membership 정합성
- booking 후속 처리 정합성
- PG 호출 중복 방지
- webhook 중복 반영 방지
- state lock 직렬화 동작

---

## 2. 공통 검증 포인트

모든 교차 충돌 테스트에서 아래를 공통 확인한다.

### payment
- 최종 status가 기대 상태인지
- 잘못된 상태 전이가 없는지
- pgPaymentKey / pgOrderId / rawResponse 정합성

### membership
- 중복 생성되지 않는지
- REFUNDED 중복 반영 없는지
- membershipId 연결이 꼬이지 않는지

### booking
- 미래 예약 취소 1회만 반영되는지
- 이미 취소된 예약을 중복 취소하지 않는지

### operation / webhook log
- payment_operation 상태가 맞는지
- payment_webhook_log 중복 저장 없는지
- transmissionId 재전송이 skip 되는지

### 외부 호출
- PG approve 최대 1회
- PG cancel 최대 1회
- PG getPayment 검증 호출 기대 범위 내인지

---

## 3. approve vs webhook completed

### 케이스 1
approve API와 webhook PAYMENT_COMPLETED가 동시에 들어온다.

#### 준비 조건
- payment status = READY
- 같은 paymentId
- webhook payload = completed
- PG 조회 결과 = completed

#### 기대 결과
- 최종 payment status = COMPLETED
- membership 1건만 생성
- approve 성공 응답 또는 webhook 성공 반영 중 하나만 실제 반영
- webhook 또는 approve 한쪽은 skip 또는 재사용/중복 차단
- PG approve 호출은 최대 1회

#### 검증 포인트
- membership count = 1
- payment status = COMPLETED
- payment_operation(APPROVE) = SUCCEEDED 또는 중복 차단 상태
- webhook log 1건 저장
- payment 상태 최종 일관성 유지

---

## 4. approve vs webhook failed

### 케이스 2
approve API와 webhook PAYMENT_FAILED가 동시에 들어온다.

#### 준비 조건
- payment status = READY
- webhook payload = failed
- PG 조회 결과 = failed 또는 completed 중 하나로 명확히 세팅

#### 기대 결과
- 먼저 반영된 상태 하나만 최종 상태가 됨
- approve가 먼저 성공 완료되면 failed webhook는 skip
- failed webhook가 먼저 반영되면 approve는 READY가 아니므로 실패

#### 검증 포인트
- 최종 payment status = COMPLETED 또는 FAILED 중 하나
- 두 상태가 섞인 중간 상태 없음
- membership 생성은 COMPLETED일 때만 1건
- PG approve 호출 횟수 기대값 검증

---

## 5. refund vs webhook canceled

### 케이스 3
refund API와 webhook PAYMENT_CANCELED가 동시에 들어온다.

#### 준비 조건
- payment status = COMPLETED
- membership 연결됨
- 미래 예약 존재
- webhook payload = canceled
- PG 조회 결과 = canceled

#### 기대 결과
- 최종 payment status = CANCELED
- membership status = REFUNDED 1회만 반영
- 미래 예약 취소 1회만 반영
- refund API와 webhook 중 하나가 먼저 처리되어도 결과는 동일
- PG cancel 호출은 최대 1회

#### 검증 포인트
- payment = CANCELED
- membership = REFUNDED
- canceled booking count 기대값 일치
- refund operation = SUCCEEDED 또는 중복 차단
- webhook log 1건
- PG cancel 1회 이하

---

## 6. refund vs webhook completed

### 케이스 4
refund API와 webhook PAYMENT_COMPLETED가 동시에 들어온다.

#### 준비 조건
- payment status = COMPLETED
- webhook payload = completed

#### 기대 결과
- 이미 COMPLETED 상태에서 completed webhook는 skip
- refund가 정상 처리되면 최종 CANCELED
- completed webhook가 refund 후 들어오면 CANCELED 상태이므로 예외 또는 skip

#### 검증 포인트
- 최종 payment status = CANCELED
- membership = REFUNDED
- webhook completed는 중복 반영 안 됨
- 상태 되돌림 없음

---

## 7. approve vs refund

### 케이스 5
approve API와 refund API가 동시에 들어온다.

#### 준비 조건
- payment status = READY

#### 기대 결과
- READY 상태에서는 refund 불가
- approve가 성공하면 COMPLETED
- refund는 READY 상태 기준으로 실패하거나, approve 이후 재조회 기준으로 정책 결정
- READY에서 바로 CANCELED 되면 안 됨

#### 검증 포인트
- 최종 payment status = COMPLETED 또는 READY 유지
- CANCELED가 되지 않는지 확인
- membership 생성은 최대 1건
- PG approve 호출 기대값 검증
- PG cancel 호출 0회

---

## 8. approve vs webhook canceled

### 케이스 6
approve API와 webhook PAYMENT_CANCELED가 동시에 들어온다.

#### 준비 조건
- payment status = READY
- webhook payload = canceled
- PG 조회 결과는 canceled 또는 not canceled 중 하나로 명확하게 설정

#### 기대 결과
- 실제 PG 상태가 canceled가 아니라면 webhook는 실패/skip
- approve 성공 가능
- 또는 PG 상태가 canceled라면 approve는 실패해야 함

#### 검증 포인트
- PG 재조회 결과에 따라 최종 상태가 정확히 반영되는지
- approve / webhook 간 상충 시 한쪽만 승리
- membership 중복 생성 없음

---

## 9. webhook completed vs webhook canceled

### 케이스 7
webhook PAYMENT_COMPLETED와 PAYMENT_CANCELED가 동시에 들어온다.

#### 준비 조건
- 동일 payment
- 서로 다른 transmissionId
- PG 조회 결과를 completed 또는 canceled로 고정

#### 기대 결과
- PG 조회 결과와 일치하는 이벤트만 반영
- 다른 이벤트는 실패 또는 skip
- 최종 상태는 PG 기준과 동일

#### 검증 포인트
- payment 최종 상태 = PG 상태
- webhook log는 각 transmissionId 기준 저장 여부 확인
- 상태가 두 번 뒤집히지 않는지 확인

---

## 10. webhook failed vs webhook completed

### 케이스 8
webhook PAYMENT_FAILED와 PAYMENT_COMPLETED가 동시에 들어온다.

#### 준비 조건
- 동일 payment
- 서로 다른 transmissionId
- PG 조회 결과를 completed 또는 failed 중 하나로 명확히 설정

#### 기대 결과
- PG 상태와 일치하는 이벤트만 반영
- completed가 최종이면 membership 생성 정책 확인 필요
- failed가 최종이면 membership 생성 없음

#### 검증 포인트
- payment 최종 상태 일관성
- membership 개수
- webhook log 저장 여부

---

## 11. 동일 webhook 재전송

### 케이스 9
같은 transmissionId로 동일 webhook를 두 번 보낸다.

#### 준비 조건
- 동일 transmissionId
- 동일 rawBody
- 동일 eventType

#### 기대 결과
- 첫 번째만 처리
- 두 번째는 duplicated transmission으로 무시
- 상태 추가 변경 없음

#### 검증 포인트
- webhook log 1건
- payment 상태 변화 1회
- PG 검증/상태 반영 1회

---

## 12. 동일 webhook 다른 payload

### 케이스 10
같은 transmissionId인데 payload만 다르게 보낸다.

#### 준비 조건
- 동일 transmissionId
- 다른 eventType 또는 다른 paymentKey/orderId

#### 기대 결과
- 현재 구조가 transmissionId 중복만 보면 두 번째는 무시될 가능성 있음
- 장기적으로는 payload hash 비교 필요 여부 검토 포인트

#### 검증 포인트
- 현재 정책상 두 번째 요청이 무시되는지
- 로그 분석 기준 확보

---

## 13. transmissionId 없음 webhook 재전송

### 케이스 11
transmissionId 없이 같은 webhook를 여러 번 보낸다.

#### 준비 조건
- transmissionId = null
- 동일 rawBody 재전송

#### 기대 결과
- 현재 구조상 중복 판별 어려움
- PG 재조회 / 상태 skip 로직으로 중복 반영이 막혀야 함

#### 검증 포인트
- completed 재전송 시 최종 상태 한 번만 반영
- canceled 재전송 시 최종 상태 한 번만 반영
- membership / booking 중복 처리 없음

---

## 14. refund vs refund

### 케이스 12
다른 idempotencyKey로 refund 요청 2개가 동시에 들어온다.

#### 준비 조건
- payment status = COMPLETED
- membership 연결됨

#### 기대 결과
- 최종 CANCELED 1회만 반영
- PG cancel 1회만 호출
- 한 요청은 성공, 다른 요청은 실패 또는 중복 차단

#### 검증 포인트
- payment = CANCELED
- membership = REFUNDED
- booking cancel 1회
- refund operation 2건 중 1건 SUCCEEDED, 1건 FAILED 기대

---

## 15. approve vs approve

### 케이스 13
다른 idempotencyKey로 approve 요청 2개가 동시에 들어온다.

#### 준비 조건
- payment status = READY

#### 기대 결과
- 최종 COMPLETED 1회만 반영
- membership 1건만 생성
- PG approve 1회만 호출

#### 검증 포인트
- payment = COMPLETED
- membership count = 1
- approve operation 2건 중 1건 SUCCEEDED, 1건 FAILED 기대

---

## 16. 동일 idempotencyKey approve 동시 요청

### 케이스 14
동일 idempotencyKey, 동일 payload로 approve 2개가 동시에 들어온다.

#### 기대 결과
- 1건 성공
- 다른 1건은 결과 재사용 또는 PROCESSING 차단
- PG approve 1회만 호출

#### 검증 포인트
- payment = COMPLETED
- membership count = 1
- operation 1건만 존재 또는 1건 기준 재사용
- response payload 복원 가능

---

## 17. 실행 우선순위 추천

교차 충돌 테스트는 아래 순서로 붙이는 걸 추천한다.

### 1순위
- approve vs webhook completed
- refund vs webhook canceled

### 2순위
- approve vs refund
- webhook completed vs webhook canceled

### 3순위
- transmissionId 없는 재전송
- 동일 transmissionId 다른 payload

---

## 18. 테스트 구현 공통 규칙

- 클래스 레벨 `@Transactional` 사용 여부 주의
- CountDownLatch 사용
- ExecutorService 2~3 thread 사용
- PG approve / cancel / getPayment mock 횟수 검증
- lock 후 재조회 전제를 깨지 않도록 실제 service 호출 기준 테스트
- 결과만 보지 말고 operation / webhook log / membership / booking까지 같이 검증

---

## 19. 한 줄 요약

교차 충돌 테스트의 핵심은  
**approve / refund / webhook가 같은 payment를 동시에 건드릴 때도 하나의 state lock 아래에서 최종 상태가 단 한 번만 일관되게 반영되는지 검증하는 것**이다.
