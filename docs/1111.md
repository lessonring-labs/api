# Payment Webhook 보안 설계

## 1. 문서 목적
이 문서는 LessOnRing 결제 웹훅 처리에서  
중복 수신, 위변조 의심 요청, 재전송, 순서 역전, 오입력 payload 등  
실서비스에서 발생 가능한 위험을 줄이기 위한  
Webhook 보안 및 검증 설계 기준을 정의한다.

---

## 2. 배경

현재 웹훅 처리 구조

- transmissionId 기반 중복 방지
- 상태 기반 멱등 처리
- webhook 로그 저장

하지만 실서비스에서는 다음과 같은 문제가 발생한다.

### 위험 요소

- 외부에서 임의 webhook 호출
- 동일 webhook 재전송
- 오래된 요청 재생 공격
- 이벤트 순서 뒤집힘
- PG 상태와 payload 불일치
- 잘못된 orderId / paymentKey

즉, webhook payload는 그대로 신뢰하면 안 된다.

---

## 3. 목표

1. 중복 webhook 처리 방지
2. 재생 공격 방어
3. payload 위변조 방어
4. 상태 전이 안전성 확보
5. PG 상태와 정합성 유지
6. 운영 추적 가능성 확보

---

## 4. 핵심 원칙

### 4.1 webhook은 신뢰하지 않는다
- payload는 참고 데이터일 뿐
- 최종 상태는 내부 정책 + PG 기준으로 결정

### 4.2 중복 방지와 보안은 분리
- 중복: transmissionId
- 보안: header / payload / 상태 / 재조회

### 4.3 상태 전이 제한 필수
- 허용된 상태 전이만 반영

### 4.4 모든 요청은 로그로 남긴다
- raw body
- headers
- 결과 상태

---

## 5. 처리 흐름

1. HTTPS 수신
2. header 검증
3. transmissionId 중복 검사
4. webhook 로그 저장
5. payload 검증
6. payment 조회
7. 상태 전이 검증
8. 필요 시 PG 재조회
9. 상태 반영 또는 무시
10. 로그 업데이트

---

## 6. Header 검증

### 필수 헤더

- transmissionId
- transmissionTime

### 검증 정책

- 누락 시 reject
- timestamp 유효성 검증
- 과거 요청 reject

### 허용 시간

- ±5 ~ 10분

---

## 7. 중복 방지

### 테이블

payment_webhook_log

### 핵심 컬럼

- provider
- transmission_id (unique)
- event_type
- order_id
- payment_key
- payload
- headers_json
- created_at

### 정책

- insert 시도
- unique 충돌 시 duplicate 처리
- duplicate는 무시

---

## 8. payload 검증

### 필수값

- orderId
- eventType

### 권장 검증

- paymentKey
- amount
- status

### 정책

- 필수값 없으면 reject
- payment 매칭 안되면 ignore
- 값 불일치 시 경고 로그

---

## 9. 내부 정합성 검증

### orderId 검증
- 내부 payment 존재 여부 확인

### paymentKey 검증
- 기존 값과 불일치 시 무시

### 금액 검증
- 불일치 시 PG 재조회

---

## 10. 상태 전이 정책

허용

READY → COMPLETED  
READY → FAILED  
COMPLETED → CANCELED

금지

CANCELED → COMPLETED  
COMPLETED → FAILED

---

## 11. PG 재조회 정책

### 재조회 대상

- PAYMENT_COMPLETED
- PAYMENT_CANCELED
- 상태 충돌 발생 시

### 목적

- 위변조 방지
- 상태 정합성 확보

### 정책

- PG 상태와 일치하면 반영
- 불일치 시 무시

---

## 12. 순서 역전 대응

문제

- FAILED → COMPLETED 순서 역전
- CANCELED 이후 COMPLETED 도착

해결

- 이벤트 순서 신뢰하지 않음
- 상태 전이 정책 우선
- 필요 시 PG 재조회

---

## 13. Lock 전략

### 목적
동일 orderId 동시 처리 방지

### key

order:webhook:{orderId}

---

## 14. 처리 정책

### PAYMENT_COMPLETED

- 상태 READY일 때만 처리
- PG 재조회 후 COMPLETED 반영
- 이미 COMPLETED면 무시
- CANCELED면 무시

---

### PAYMENT_FAILED

- READY 상태일 때만 처리
- 필요 시 PG 재조회
- COMPLETED 상태면 무시

---

### PAYMENT_CANCELED

- COMPLETED 상태일 때만 처리
- PG 재조회 후 반영

---

## 15. HTTP 응답 정책

- 정상 처리 → 200
- duplicate → 200
- 잘못된 요청 → 400
- 검증 실패 → 400 / 401

---

## 16. Outbox 연계

웹훅 처리 후 이벤트는 outbox에 저장

### 이유

- 외부 장애 분리
- 트랜잭션 안정성 확보

---

WEBHOOK_REJECTED  
WEBHOOK_INVALID_PAYLOAD  
WEBHOOK_INVALID_STATE  
WEBHOOK_REFETCH_REQUIRED  
WEBHOOK_APPLIED  
WEBHOOK_IGNORED

---

## 18. 테스트 시나리오

### 중복

- 동일 transmissionId → 무시

### header

- 누락 → reject
- 시간 초과 → reject

### payload

- orderId 없음 → reject
- paymentKey mismatch → 무시

### 상태

- 잘못된 상태 전이 → 무시

### 재조회

- payload ≠ PG → 무시

---

## 19. 도입 순서

### 1단계

- transmissionId unique
- header 검증
- payload 검증

### 2단계

- 상태 전이 정책
- PG 재조회

### 3단계

- lock 적용
- verifier 구조 분리

---

## 20. 최종 요약

Webhook 보안은

중복 방지 + header 검증 + payload 검증 + 상태 전이 제한 + PG 재조회

조합으로 구현하며

webhook payload는 절대 신뢰하지 않고  
항상 내부 상태와 비교 후 반영한다.
